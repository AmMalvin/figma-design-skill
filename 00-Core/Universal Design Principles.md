# Human-Centered Principles

## Principle 05. Design for Humans First

### Statement

Every design decision must prioritize human needs before technology, business convenience, or visual preference.

### Intent

Design exists to help people accomplish goals efficiently, confidently, and comfortably.

Interfaces should reduce effort rather than create it.

### Rationale

Technology changes.

Business requirements evolve.

People remain the center of every successful product.

Design that ignores human behavior inevitably creates friction.

### Rules

- Begin every project with user goals.
- Understand the context of use.
- Design around user tasks.
- Respect human limitations.
- Reduce unnecessary effort.
- Consider emotional responses.

### AI Instructions

Always identify:

- Who is using the product.
- What they want to achieve.
- What prevents them from succeeding.
- What success looks like.

Never optimize visuals before understanding users.

---

## Principle 06. Solve Problems Before Designing Interfaces

### Statement

A beautiful interface that fails to solve the user's problem is a failed design.

### Intent

Design outputs must exist to solve measurable problems.

### Rules

- Define the problem first.
- Verify the problem exists.
- Understand root causes.
- Design solutions, not screens.
- Measure outcomes.

### AI Instructions

Before creating any UI, produce:

- Problem Statement
- Business Objective
- User Objective
- Success Metrics

Only then proceed to design.

---

## Principle 07. Design for Context

### Statement

Interfaces must adapt to the environments in which they are used.

### Intent

Good design changes based on where, when, and how users interact.

### Rules

Consider:

- Device
- Screen size
- Lighting
- Internet quality
- User attention
- Physical environment
- Time pressure

### AI Instructions

Always ask:

Where will this experience be used?

How will users interact?

What environmental constraints exist?

---

## Principle 08. Respect User Time

### Statement

Every interaction should minimize unnecessary effort.

### Intent

Users value efficiency.

Every unnecessary click, tap, scroll, or decision reduces experience quality.

### Rules

Reduce:

- Clicks
- Cognitive load
- Waiting
- Searching
- Manual input

Increase:

- Automation
- Smart defaults
- Predictability
- Efficiency

### AI Instructions

Always evaluate whether a workflow can be simplified without reducing clarity.

# Visual Design Principles

## Principle 17. Hierarchy Before Decoration

### Statement

Visual hierarchy shall communicate importance before aesthetics.

### Intent

Users should immediately understand what deserves attention without reading every element on the screen.

### Rationale

Strong hierarchy improves scanning, comprehension, and decision making. Decorative styling must never reduce clarity.

### Rules

- Establish one clear primary focal point.
- Use size, weight, spacing, and contrast intentionally.
- Reduce visual competition.
- Every page must have a predictable reading order.

### AI Instructions

Before styling any interface, identify:

- Primary action
- Secondary action
- Supporting information
- Background information

Design the hierarchy before applying colors or visual effects.

---

## Principle 18. Consistency Creates Trust

### Statement

Consistency improves usability, reduces learning time, and strengthens confidence.

### Intent

Users should recognize familiar patterns instead of learning new ones on every screen.

### Rationale

Predictable interfaces reduce cognitive effort and improve efficiency.

### Rules

Maintain consistency across:

- Components
- Spacing
- Typography
- Color usage
- Icons
- Motion
- Content structure
- Interaction patterns

### AI Instructions

Before creating a new solution, verify whether an approved solution already exists.

---

## Principle 19. Color Must Communicate Meaning

### Statement

Color is functional before decorative.

### Intent

Color should communicate hierarchy, status, feedback, and meaning.

### Rationale

Color improves recognition when used consistently. Misuse creates confusion and accessibility issues.

### Rules

Never use color without purpose.

Reserve color for:

- Actions
- Status
- Feedback
- Categories
- Focus
- Selection

Never rely on color alone to communicate information.

### AI Instructions

Use semantic color tokens instead of raw color values.

---

## Principle 20. Typography Creates Structure

### Statement

Typography is the primary tool for organizing information.

### Intent

Users should understand relationships between content through typography alone.

### Rationale

Typography influences readability, hierarchy, and comprehension more than decorative styling.

### Rules

Typography must define:

- Page hierarchy
- Section hierarchy
- Reading flow
- Emphasis
- Content grouping

Limit the number of text styles.

### AI Instructions

Always use typography tokens.

Never create one-off text styles.

---

## Principle 21. Spacing Is Communication

### Statement

Whitespace defines relationships between interface elements.

### Intent

Spacing should improve comprehension rather than simply fill empty space.

### Rationale

Users naturally associate elements placed close together and separate unrelated information.

### Rules

Spacing should communicate:

- Grouping
- Separation
- Priority
- Rhythm

Spacing values must follow the spacing system.

### AI Instructions

Never apply arbitrary spacing.

Always use spacing tokens.

---

## Principle 22. Alignment Creates Order

### Statement

Alignment improves readability and predictability.

### Intent

Users should perceive a structured interface rather than isolated objects.

### Rationale

Misalignment increases visual noise and slows information processing.

### Rules

Align content using:

- Grid systems
- Layout grids
- Auto Layout
- Baselines

Avoid inconsistent positioning.

### AI Instructions

Validate alignment before considering any layout complete.

---

## Principle 23. Simplicity Is a Design Requirement

### Statement

Every interface should contain only what users need.

### Intent

Remove unnecessary complexity without reducing capability.

### Rationale

Simple interfaces reduce cognitive load and improve confidence.

### Rules

Remove:

- Redundant controls
- Duplicate information
- Decorative distractions
- Unnecessary interactions

Keep only elements that support user goals.

### AI Instructions

Review every element and justify its existence.

If an element has no measurable purpose, remove it.

---

## Principle 24. Motion Must Support Understanding

### Statement

Animation exists to communicate change, not decoration.

### Intent

Motion should reinforce spatial relationships and system feedback.

### Rationale

Purposeful animation helps users understand transitions and state changes.

### Rules

Motion should:

- Explain
- Guide
- Confirm
- Emphasize

Avoid excessive or distracting animation.

### AI Instructions

Every animation must answer:

- What changed?
- Why did it change?

# Interaction Design Principles

## Principle 25. Recognition Before Recall

### Statement

Interfaces shall minimize memory load by making available actions, options, and information visible whenever practical.

### Intent

Users should recognize available actions instead of remembering them.

### Rationale

Recognition reduces cognitive effort and improves task completion.

### Rules

- Prefer visible controls.
- Display important information.
- Keep navigation predictable.
- Avoid hidden interactions.
- Reveal advanced actions progressively.

### AI Instructions

Never require users to memorize workflows.

Design interfaces that expose functionality naturally.

---

## Principle 26. Immediate Feedback

### Statement

Every user action must produce timely and meaningful feedback.

### Intent

Users should always understand the result of their actions.

### Rationale

Feedback builds confidence and reduces uncertainty.

### Rules

Provide feedback for:

- Clicks
- Taps
- Keyboard actions
- Loading
- Success
- Failure
- Validation
- Progress

### AI Instructions

Never leave users wondering whether the system received their action.

---

### Intent

Interfaces should reduce opportunities for human error.

### Rules

- Validate early.
- Disable impossible actions.
- Use smart defaults.
- Confirm destructive actions.
- Explain requirements before submission.

### AI Instructions

Identify potential mistakes before designing the interface.

Design safeguards into the workflow.

---

# Accessibility Principles

## Principle 33. Accessibility Is a Requirement, Not a Feature

### Statement

Accessibility is a mandatory quality requirement that must be integrated into every stage of the design process.

### Intent

Every product should be usable by the widest practical audience regardless of ability, environment, or technology.

### Rationale

Accessibility improves usability for everyone. It is not a separate mode of design but a core design responsibility.

### Rules

- Consider accessibility from project kickoff.
- Never postpone accessibility until QA.
- Meet WCAG AA or higher where applicable.
- Include accessibility in every design review.

### AI Instructions

Accessibility must influence every design decision.

Never treat accessibility as optional.

---

## Principle 34. Never Rely on Color Alone

### Statement

Information must never depend solely on color.

### Intent

Users with color vision deficiencies should receive the same information as everyone else.

### Rationale

Status, errors, warnings, and success indicators require multiple visual cues.

### Rules

Support color with:

- Icons
- Labels
- Patterns
- Helper text
- Shape changes

### AI Instructions

Whenever color communicates meaning, provide at least one additional indicator.

---

- Logical tab order

- Skip navigation
- Focus trapping in dialogs

### AI Instructions

Every interactive component must define keyboard behavior.

# Engineering Principles

## Principle 45. Design for Implementation

### Statement

Every design must be feasible to implement using modern engineering practices.

### Intent

Design should reduce ambiguity and accelerate development.

### Rationale

A design system succeeds only when designers and developers share the same understanding of components, behaviors, and constraints.

### Rules

- Design with implementation in mind.
- Prefer established patterns over custom solutions.
- Minimize unnecessary complexity.
- Consider engineering effort during decision making.
- Validate feasibility before approval.

### AI Instructions

Before introducing a new pattern, evaluate implementation cost, maintainability, and long-term support.

---

## Principle 46. Semantic Tokens Before Raw Values

### Statement

Interfaces must use semantic design tokens instead of hardcoded values.

### Intent

Semantic tokens improve scalability, theming, and long-term maintenance.

### Rules

Never reference:

- Hex colors
- Pixel spacing
- Radius values
- Shadows
- Font sizes

Instead reference semantic tokens.

### AI Instructions

Whenever a raw value appears, replace it with the appropriate semantic token.

---

## Principle 47. Components Are Contracts

### Statement

Every component represents a reusable contract between design and engineering.

### Intent

Components must behave consistently regardless of where they appear.

### Rules

Every component must define:

- Purpose
- Anatomy
- Properties
- Variants
- States
- Behaviors
- Accessibility
- Usage
- Limitations

### AI Instructions

Never publish undocumented components.

---

## Principle 48. Auto Layout Is the Default

### Statement

Every component and layout should support responsive behavior through Auto Layout.

### Intent

Responsive behavior should be built into the component rather than added later.

### Rules

Use:

- Hug
- Fill
- Fixed sizing only when required

Avoid manual positioning whenever possible.

### AI Instructions

Reject layouts that cannot adapt to changing content.

---

## Principle 49. Variables Drive Consistency

### Statement

Variables provide the single source of truth for reusable design values.

### Intent

Variables simplify theming, scaling, and maintenance.

### Rules

Store:

- Colors
- Spacing
- Radius
- Borders
- Opacity
- Motion
- Elevation

inside variables.

### AI Instructions

Do not create duplicate variables for identical purposes.

---

## Principle 50. Naming Must Be Predictable

### Statement

Every design asset requires a clear, scalable naming convention.

### Intent

Naming should improve discoverability and reduce ambiguity.

### Rules

Names must be:

- Consistent
- Descriptive

Follow repository naming conventions exactly.

- Skip accessibility validation.

- Create duplicate components.
- Create duplicate variables.
- Hardcode reusable design values.
- Ignore platform conventions.
- Introduce inconsistent naming.
- Publish undocumented assets.
- Remove reusable assets without a migration strategy.

The AI must always:

- Think before designing.
- Reuse before creating.
- Validate before publishing.
- Document while building.
- Use semantic variables.
- Follow repository naming conventions.
- Maintain a single source of truth.
- Consider scalability.
- Consider maintainability.

# Principle Conflict Resolution

When two principles conflict, resolve them in the following order:

1. User safety
2. Accessibility
3. User needs
4. Business objectives
5. Product strategy
6. System integrity
7. Consistency
8. Scalability
9. Maintainability
10. Visual refinement

Higher priorities always override lower priorities.

□ Does this solve a real user problem?

□ Is the workflow understandable?

□ Is the experience efficient?

□ Is the interface predictable?

---

□ Is WCAG compliance satisfied?

□ Is contrast sufficient?

□ Can keyboard users complete the task?

□ Are touch targets appropriate?

□ Does the interface support assistive technologies?

---

## Visual Design

□ Is hierarchy clear?

□ Is spacing consistent?

□ Is typography consistent?

□ Is alignment correct?

□ Is color meaningful?

---

## Design System

□ Are semantic tokens used?

□ Are variables used correctly?

□ Are existing components reused?

□ Are naming conventions followed?

□ Are patterns consistent?

---

## Documentation

□ Is the purpose documented?

□ Is usage documented?

□ Are variants documented?

□ Are accessibility notes included?

# Definition of Compliance

A design artifact is considered compliant only when:

- Every applicable principle has been satisfied.
- Required documentation is complete.
- Validation has passed.
- Accessibility requirements are met.
- Engineering feasibility has been confirmed.
- The design strengthens, rather than weakens, the design system.

Failure to satisfy any mandatory requirement prevents publication.

# Closing Statement

These Universal Design Principles form the governing constitution of this Design System Skill.

Every module, workflow, component, pattern, variable, token, documentation page, and future contribution inherits these principles by default.

No module may contradict these principles.

Where a module introduces additional rules, those rules extend these principles and never replace them.

The integrity of the design system depends on consistent application of these principles across every stage of discovery, design, validation, documentation, implementation, maintenance, and governance.
